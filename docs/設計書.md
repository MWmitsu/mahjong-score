# 麻雀スコア管理アプリ 設計書

- 対象アプリ：麻雀スコア（Web版・PWA） `https://mwmitsu.github.io/mahjong-score/`
- リポジトリ：`MahjongScoreWeb`（`github.com/MWmitsu/mahjong-score`）
- 最終更新：2026-06-21

> この文書は「何のためのアプリか」「データをどう持つか」「点数と精算をどう計算するか」を1か所にまとめたものです。とくに **合計（卓の点数・精算額）が0にならないことがある理由**（§5.6）を、実際の数値例つきで説明します。

---

## 1. アプリの目的

1ゲーム（半荘）が終わるたびに **最終的な持ち点（粗点）を入力するだけ** で、
自動的にポイント計算・累計成績・人物別成績・ランキングを出す、麻雀の成績管理アプリ。

- 3人麻雀・4人麻雀の両方に対応（種別ごとに集計を分ける）
- 1つの「部屋」＝1日分の成績表（複数の半荘を行として積む）
- チップ・役満祝儀・レート（円換算）まで含めて精算できる
- Googleログインで自動クラウド保存・複数端末で同期
- インストール不要（ホーム画面アプリ／PWAとしても使える）

## 3. 用語

| 用語 | 意味 |
|---|---|
| 部屋（成績表） | 1日分の記録のまとまり。複数の半荘を持つ |
| 半荘（はんちゃん） | 1ゲーム。成績表の1行 |
| 粗点（そてん） | 半荘終了時の各自の持ち点（例：25000） |
| 返し点 | 精算の基準点（例：30000）。オカの原資になる |
| ウマ | 順位ごとの増減ポイント（例：20 / 10 / −10 / −20） |
| オカ | 1位が総取りするボーナス。（返し点−持ち点）×人数分 |
| 飛び（箱割れ） | 持ち点が0点以下になること |
| 箱下（はこした）なし | 精算では0点未満を0点として扱う（マイナス分は移動しない） |
| 役満祝儀 | 役満が出たときのご祝儀（チップ／ポイント／円のいずれか） |
| チップ | 半荘とは別に付ける枚数（成績表の最終行で入力） |
| レート | ポイント→円の換算率（円 / 1pt） |
| 精算 | 各自の最終的な受け払い（円） |

## 4. データモデル

localStorage に次の1つのJSONを保存します。

```
{
  schemaVersion: 3,
  players:  [ プレイヤー ],
  rules:    [ ルール ],
  sessions: [ 部屋（成績表） ]
}
// 旧 rooms / matches コレクションは撤去済み（読み込み・復元時に自動で除去し schemaVersion 3 へ移行）
```

### プレイヤー
```
{ id, name, memo, isActive, isSample, isDeleted, createdAt, updatedAt }
```

### ルール
```
{ id, name, mahjongType('three'|'four'),
  initialScore, returnScore,            // 持ち点・返し点
  hasOka, okaPoint,                     // オカ
  umaPoints[],                          // 順位ウマ（人数分）
  pointPer1000,                         // 1000点あたりのポイント（通常 1）
  hasTobiBonus, tobiBonusPoint, tobiPaymentType,   // 飛び賞
  bustRule, noNegativeSettlement,       // 飛び判定・箱下なし
  chipUnitAmount, pointToYenRate, chipPointIncludedInMainRanking,
  yakumanShugiType('none'|'chip'|'point'|'yen'),   // 役満祝儀の処理
  roundingRule, isActive, isSample, isDefault, isDeleted, ... }
```
- `isDefault:true` … 自動で作られる初期ルール（クラウド判定で「実データ」に数えない）

### 部屋（成績表 / session）
```
{ id, name, date, mahjongType, ruleSetId, ruleName,
  rate,            // レート（円/pt）
  chipUnit,        // チップ単価（作成時にルールから固定）
  shugiType,       // 役満祝儀の処理（作成時に固定）
  playerIds[],     // 登録メンバー（席数以上でもよい）
  hanchans[],      // 半荘の配列（下記）
  chips{ playerId: 枚数 },   // 最終行のチップ
  isSample, isDeleted, createdAt, updatedAt }
```

### 半荘（hanchan）
```
{ id,
  playerIds[],     // その半荘の出場者（抜け番対応。ちょうど席数）
  raws{ playerId: 粗点 },
  results[],       // 計算済みの結果スナップショット（下記）
  shugi,           // 役満祝儀（任意）
  createdAt }
```
- `results` は **入力確定時に一度だけ計算して保存** します（表示・集計はこれを読む）。
  そのためルールを後から変更しても、既存の半荘の点数はさかのぼって変わりません（スナップショット方式）。

### 結果（results の各要素）
```
{ playerId, rawScore, rank, basePoint, totalPointWithoutChip, isBusted, ... }
```

### 役満祝儀（shugi）
- 新形式：`{ values: { playerId: 増減 } }`（各自に直接入力・合計0）
- 旧形式：`{ winnerId, amount }`（和了者＋各他家の枚数。読み込み時に出場者へ換算）

## 5. 計算のしくみ（核心）

### 5.1 1半荘の点数計算

各プレイヤーの **ポイント** を次の順で足し合わせます（`domain.js` の `calculate`）。

1. **精算粗点** ＝ `max(粗点, 0)`（箱下なしのとき。0点未満は0点扱い）
2. **基礎点** ＝ `(精算粗点 − 返し点) ÷ 1000 × pointPer1000`
3. **オカ** ＝ 1位のみに `(返し点 − 持ち点) ÷ 1000 × 人数`
4. **ウマ** ＝ 順位に応じて `umaPoints[順位−1]`
5. **飛び賞** ＝ 飛びがあれば（§5.3）
6. **丸め** ＝ ルールの小数点処理（なし／四捨五入／切上げ／切下げ／五捨六入）

**例（4人麻雀 標準：25000持ち・30000返し・ウマ 20/10/−10/−20・オカ20）**

| 席 | 粗点 | 基礎点 | オカ | ウマ | 合計pt |
|---:|---:|---:|---:|---:|---:|
| 甲（1位） | 40000 | +10 | +20 | +20 | **+50** |
| 乙（2位） | 30000 | 0 | — | +10 | **+10** |
| 丙（3位） | 20000 | −10 | — | −10 | **−20** |
| 丁（4位） | 10000 | −20 | — | −20 | **−40** |
| **卓合計** | 100000 | | | | **0** |

**トップ＝残り（卓合計を必ず0に）**：ポイント変換では **1位（トップ）のptを「下位プレイヤーの合計の符号反転」** として求めます。これにより、箱下やオカのあまりを **トップが負い、卓の合計ポイントは常に 0** になります。全員が0点以上のゼロサム卓では、この処理をしても値は変わりません。

### 5.2 順位の決め方

- 粗点の大きい順。
- **同点**のときは自動で決めず、入力時にポップアップで **上位から順に手動選択**（各入力の `tieBreak` に反映）。

### 5.3 飛び（箱割れ）

- 半荘入力で **0点以下の人がいるときだけ**「飛びましたか？」を確認します（手動判定）。
- 飛んだと記録すると 💥 が付き、人物別成績の「トビ回数」に加算されます。
- **飛び賞あり**のルールでは、さらに「飛ばした人」を選び、飛び賞ポイントが移動します
  （例：飛ばした人 +10 / 飛んだ人 −10）。飛び賞なしでも飛び自体は記録します。

### 5.4 役満祝儀

- ルールの「役満祝儀の処理」で **チップ／ポイント／円** のいずれか（またはなし）。
- どの種類でも **各自の増減を直接入力し、合計が0** になるように入れます（例：和了者 +2、他 −1 ずつ）。合計が0でないと登録できません。
- 役満祝儀は **精算にだけ反映**し、順位点（ウマ・オカ）には影響しません。

### 5.5 精算（円）

各自の精算額を次で求めます（`sheets.js` の `playerTotals`）。

```
精算(円) = round(合計ポイント × レート) + チップ枚数 × チップ単価 + 役満祝儀(円換算)
```

- 役満祝儀の円換算：チップ型＝枚数×単価、ポイント型＝pt×レート、円型＝そのまま。
- チップは成績表の最終行で **各自の枚数** を入力します。

### 5.6 ★ 合計とエラー表示（合計が0にならない/ずれるとき）

**卓の合計ポイントは常に 0 になります。** ポイント変換で **トップ（1位）が残り（下位合計の符号反転）を負う** ため、箱下やオカのあまりはトップに集約され、各半荘の合計は必ず0になります（§5.1）。

そのうえで、**入力が食い違っている行は「赤い！」でエラー表示** します。ポイント自体はトップ調整で0に見えても、元の入力がおかしい場合に気づけるようにするためです。

| エラーの種類 | 条件 | 表示 |
|---|---|---|
| ポイントの合計ずれ | 半荘のポイント合計 ≠ **0** | その **半荘の行**に赤い！ |
| 粗点の合計ずれ | 半荘の粗点合計 ≠ **人数 × 初期持ち点** | その **半荘の行**に赤い！ |
| チップの合計ずれ | チップの合計 ≠ **0** | **チップ行**に赤い！ |

> これらの赤い！は、**データ管理 →「エラー表示（！マーク）」で項目ごとにオン/オフ**できます（既定はすべてオン・全端末で同期）。

**入力補助（残り1人を自動計算）**

- **粗点**：出場者のうち **1人を残して入力すると、最後の1人を自動計算**（合計＝人数×持ち点）。
- **チップ**：同様に **1人を残して入力すると、最後の1人を自動計算**（合計＝0）。
- **タイミング**：入力途中では走らず、**その欄の入力を終えて次の欄へ移った（フォーカスが外れた）とき**に反映します。最後の1人を空欄のまま「確定」した場合は確定時にも自動計算します。
- 自動で入った値は **修正可**。手で直して合計が合わなくなった行は、上表の赤い！で分かります。

**箱下（はこした）の扱い**

0点未満は精算で0点扱い（箱下なし）です。以前はその「沈んだ分」が卓合計のプラスずれになっていましたが、現在は **トップがその残りを負う** ため卓合計は0に収まります。

**精算（円）もゼロサム**：レート換算の円も**各自を符号対称に丸め、丸め残差はトップ（合計ポイント最大）が負う**ため、**チップが±0なら精算の合計は必ず0円**になります。合計が0にならないのは「チップの合計が0でない」ぶんだけで、それは③の赤い！で分かります。

## 6. 画面と操作

| 画面 | 役割・主な操作 |
|---|---|
| ホーム | 「成績をつける（部屋）」＋各画面への入口 |
| 部屋一覧 | 成績表の一覧・新規作成（種別→ルール→レート→メンバー） |
| 成績表 | 表（行=半荘・列=プレイヤー）。上部にレート・自動保存バッジ、最終行にチップ、下に精算。「＋半荘を入力」 |
| 半荘入力 | 出場者選択（抜け番時）→粗点入力→0点以下・同点だけ確認。役満祝儀は任意 |
| プレイヤー別成績 | 期間・種別・部屋で絞り、累計グラフ・順位率・連勝連敗・トビ・役満回数 |
| ランキング | 種別ごとに合計pt・平均・トップ率・平均順位・ラス回避率・精算額(参考) |
| ルール管理 | 3麻/4麻別の作成・編集・複製・削除（使用中は無効化へ誘導） |
| プレイヤー管理 | 追加・編集・有効/無効・削除（履歴ありは無効化へ誘導） |
| データ管理 | クラウド同期・JSONバックアップ/復元・CSV出力・サンプル/全削除 |

### 抜け番（3麻で4人以上／4麻で5人以上）
- 部屋に **席数より多いメンバー** を登録できます。
- 半荘ごとに **出場者（ちょうど席数）を選択**（初期選択は前回と同じ人）。
- 出ていない人はその半荘の表で「—」。成績は **実際に打った半荘だけ** で集計します。

## 7. 保存とクラウド同期

- 入力のたびに localStorage へ自動保存。ログイン時は Firestore（`users/{uid}`）へも自動保存。
- ログイン方式：`signInWithPopup`（全機種共通。iOSのPWAでリダイレクトが戻らない不具合を回避）。
- 端末をまたぐ整合：
  - クラウドが空 → ローカルを上げる
  - ローカルが空 → クラウドを取り込む
  - 両方にデータ → 「クラウドを使う／この端末を使う」を確認
- 「実データ有無」の判定は **プレイヤー／部屋／ユーザー作成ルール** で行い、**自動生成の初期ルールは数えません**（新端末でクラウドを誤って上書きしないため）。
- 未送信の編集がある間は他端末の反映を一時見送り、編集の消失を防ぎます。
- `file://`（ファイル直開き）ではクラウドは使えません（オンラインURLを案内）。

## 8. バックアップ・復元・CSV

- **JSONバックアップ**：全データを書き出し。復元で全置換（配列でない壊れたファイルは弾く）。
- **CSV出力**：半荘明細（日付・部屋・種別・半荘・プレイヤー・順位・粗点・ポイント・飛び・役満）。抜け番は出場者のみ行に出力。
- **サンプル削除**：サンプルだけ削除。ただし自分の部屋で使っているサンプルプレイヤーは残す。
- **全削除**：初期ルールだけ残して全消去。

## 9. 確定した設計の約束事

1. 箱下なし＝精算粗点は `max(粗点,0)`。**トップ（1位）が残りを負い、卓の合計ポイントは常に0**。順位は粗点で決定。
2. 1部屋＝1種別（3麻/4麻）固定。成績・ランキングは常に種別を1つ選択（合算しない）。
3. レート・チップ単価・役満祝儀の処理は **部屋作成時にスナップショット**（後からルールを変えても既存部屋は不変）。
4. ルール編集は過去の対局に **さかのぼらない**（results は保存済みの値を使う）。
5. 削除は原則 **論理削除**。使用中のルール・履歴ありのプレイヤーは物理削除せず「無効化」へ誘導。
6. ポップアップは **飛び・同点のときだけ**。通常は粗点を入れるだけ。

## 10. 既知の挙動・制限・今後の選択肢

| 項目 | 現状 | 今後の選択肢 |
|---|---|---|
| 卓の合計ポイント | トップが残りを負い常に0。粗点の入力ずれは赤い！で表示（§5.6） | 「トップに寄せず均等割り」等の別方式を追加可 |
| チップの合計 | 残り1人を自動計算し0に。合計≠0は赤い！ | 「合計0を強制（保存不可）」に変更可 |
| 精算(円) | トップが端数を負いゼロサム。チップ非均衡ぶんのみ残る（！で表示） | 端数の負担先を最下位に変更可 |
| ルール変更の遡及 | しない（スナップショット） | 「この部屋を新ルールで再計算」機能を追加可 |
| 削除済み部屋 | 論理削除で内部に残る | 「完全消去」導線を追加可 |
| 2端末の同時編集 | 未送信優先で消失は防止（最終書き込み優先） | 変更のマージは将来課題 |

## 11. 品質保証

- **自己テスト**：`MJ.selftest.run()` … 点数計算の主要ケースを検証（現在 **20/20**）。
- **多エージェント監査（2026-06-21）**：7観点・約36エージェントでコード全体を静的監査し、確定した実バグ **22件を修正・再検証**。主な修正：
  - 新端末ログイン時にクラウド全データを上書きする事故（初期ルール誤判定）
  - 使用中ルール／履歴ありプレイヤーの物理削除
  - 同期競合による未送信編集の消失
  - 粗点の不正値（NaN/指数）混入
  - 飛び賞オフでも飛びを記録
  - 四捨五入/五捨六入の符号非対称（卓合計ズレ）
  - 抜け番部屋での旧形式役満祝儀の誤配分
  - 復元時の壊れデータ／`#sheet` リロード時の部屋復元 ほか
- **動的テスト**：プレビュー実機で各機能を再現テスト（全画面エラーなし・コンソールエラー0）。
